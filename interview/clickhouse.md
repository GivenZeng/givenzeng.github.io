
# 一、基础原理类
## 1. ClickHouse 是什么？适合什么场景？
**答**：
ClickHouse 是 Yandex 开源的 **列式存储、分布式、面向 OLAP** 的数据库。
- 适合：**海量数据、少更新、多查询、聚合分析、宽表、大吞吐**。
- 不适合：高频小事务、高并发点查、频繁更新删除。

## 2. 为什么 ClickHouse 快？
**答**：
1. **列式存储**：只读取需要的列，IO 极小。
2. **数据压缩**：列数据相似性高，压缩比极高。
3. **向量化执行**：CPU 指令级并行，批量处理。
4. **索引**：主键索引、跳数索引（minmax、set、bloomfilter）。
5. **多核并行**：充分利用 CPU 多核。
6. **预聚合**：物化视图、Summary 结构。
7. **分布式架构**：分片+副本，水平扩展。

## 3. 行存 vs 列存区别？
**答**：
- 行存：一行数据连续存储 → 适合**点查、事务、小表**。
- 列存：一列数据连续存储 → 适合**OLAP、聚合、宽表**。

# 二、存储引擎类
## 4. 最常用引擎：MergeTree 家族
**答**：
核心是 **MergeTree**，其他都是它的变种：
- **MergeTree**：基础，支持主键、分区、索引。
- **ReplacingMergeTree**：按主键去重（合并时才去重）。
- **SummingMergeTree**：按维度预聚合求和。
- **AggregatingMergeTree**：预聚合，配合聚合函数。
- **CollapsingMergeTree**：用 Sign 位实现数据折叠。
- **VersionedCollapsingMergeTree**：带版本折叠。

## 5. MergeTree 存储结构？
**答**：
按 **分区（Partition）** 组织，每个分区内：
- 按 **主键排序** 存储
- 生成 **索引文件（.idx）**
- 生成 **列数据文件（.bin）**
后台异步 **合并（merge）** 小片段。

## 6. ClickHouse 主键和 MySQL 主键区别？
**答**：
- MySQL：主键是**唯一约束**，必须唯一。
- ClickHouse：主键是**排序键**，**不保证唯一**，只决定数据物理顺序。
去重靠 ReplacingMergeTree / CollapsingMergeTree。

# 三、索引与查询
## 7. ClickHouse 索引类型？
**答**：
1. **主键索引**：排序索引，加速范围查询。
2. **跳数索引（Skipping Index）**：
   - minmax
   - set
   - bloomfilter
   - ngrambf
作用：**跳过不满足条件的数据块**，不是精确索引。

## 8. 为什么有时候加了索引反而更慢？
**答**：
- 索引本身也要占 IO、CPU。
- 过滤性差、基数低的列不适合建索引。
- 小表、全表扫描更快时，索引无用。

## 9. ClickHouse 为什么不建议用 `SELECT *`？
**答**：
列式数据库，`*` 会读取**所有列**，IO 爆炸，严重违背设计初衷。
只查需要的列。

# 四、SQL 与语法坑
## 10. NULL 处理注意什么？
**答**：
- ClickHouse 对 NULL 支持有限，很多函数**不兼容 NULL**。
- 建议用 **默认值**（0、''）代替 NULL。
- 可用 `Nullable`，但会**降低性能**。

## 11. `IN` 和 `JOIN` 哪个好？
**答**：
- 小集合用 `IN` / `GLOBAL IN`。
- 大表关联用 **Join**，但 ClickHouse Join 性能一般，**优先宽表预聚合**。

## 12. ClickHouse 支持事务吗？
**答**：
**不支持完整事务**，只支持**单块写入原子性**，没有回滚、没有隔离级别。

# 五、分布式与架构
## 13. 分布式表原理？
**答**：
- 分布式表 **不存数据**，是**逻辑视图**。
- 数据存在本地表（如 `MergeTree`）。
- 查询时：分布式表把请求下发到各节点 → 各节点计算 → 结果汇总返回。

## 14. 分片与副本？
**答**：
- **分片（Shard）**：水平切分数据，提高吞吐。
- **副本（Replica）**：同分片多副本，保证高可用、容错。
一般用 **分片+副本+ZooKeeper/ClickHouse Keeper**。

## 15. 数据怎么分发到分片？
**答**：
通过 **分片键**：
- `rand()`
- 哈希（intHash64、cityHash64）
- 业务字段（user_id、event_date）
尽量让数据**均匀分布**。

# 六、优化面试高频
## 16. ClickHouse 查询慢怎么优化？
**答**（标准八股）：
1. **只查需要的列**，禁止 `SELECT *`。
2. **加合理分区**，按时间/用户分区。
3. **用好主键**，把过滤条件放主键前面。
4. **建跳数索引** 加速过滤。
5. **大表拆小分区**，避免大分区。
6. **预聚合**：SummingMergeTree、物化视图。
7. **避免复杂 JOIN**，尽量宽表。
8. **控制并发**，防止打爆CPU。
9. **数据类型最小化**：用 UInt32 不用 UInt64。
10. **关闭不必要的日志/统计**。

## 17. 物化视图是什么？
**答**：
- 是**预计算结果表**，不是简单视图。
- 写入源表时自动把数据写入物化视图。
- 适合**固定口径报表、大屏、统计**。

## 18. ClickHouse 为什么不适合高频更新？
**答**：
更新是**重写整个数据片段**，后台还要合并，非常昂贵。
设计上就是 **写一次、读多次**。

# 七、运维与生产问题
## 19. 数据为什么没立刻查到？
**答**：
写入后先到 **内存表（MemoryBuffer）**，异步落盘。
可：
- `SYSTEM FLUSH TABLE`
- 调整 `async_insert`

## 20. 副本不同步怎么办？
**答**：
- 检查 ZK/Keeper 状态。
- 检查网络、时钟。
- 看日志 `/var/log/clickhouse-server`。
- 用 `ALTER TABLE ... ATTACH PARTITION` 修复。

## 21. 数据膨胀、磁盘满？
**答**：
- 小片段太多，merge 跟不上。
- 删除/更新产生冗余数据。
- 解决：
  - 提高 merge 线程
  - 清理过期分区
  - 手动 optimize

3. 主键是**排序键**，不唯一。
4. 去重靠 **ReplacingMergeTree**。
