# 一、基础概念
## 1. Doris 是什么？适用场景？
**答**：
Apache Doris（原百度 Palo）是一款**分布式、高性能、实时分析型 MPP 数据库**。
- 适用：**实时数仓、OLAP 多维分析、宽表查询、高并发报表、低延迟写入查询**。
- 支持：**标准 SQL、高并发、实时写入、多维聚合**。

## 2. Doris 核心特点
**答**：
1. **兼容 MySQL 协议**，直接用 MySQL 客户端/驱动连接。
2. **MPP 架构**，多节点并行计算。
3. **列式存储 + 向量化执行**。
4. **高实时**：秒级写入、秒级查询。
5. **简单易运维**：无外部依赖（不依赖 Hadoop、ZK）。
6. **支持高并发点查 + 复杂聚合**。

---

# 二、架构类（高频）
## 3. Doris 架构组成？
**答**：
Doris 分为两层：
1. **FE（Frontend）**
   - 负责：元数据管理、SQL 解析、查询规划、集群管理。
   - 角色：Follower、Observer。
2. **BE（Backend）**
   - 负责：数据存储、SQL 执行、数据复制。
   - 所有计算、IO 都在 BE 完成。

**优点**：架构简单、无第三方依赖、扩缩容方便。

## 4. FE 的 Follower 和 Observer 区别？
**答**：
- **Follower**：参与选主、元数据写入、元数据同步。
- **Observer**：只同步元数据、**分担查询压力**，不参与选主。

## 5. Doris 数据如何分布？
**答**：
- 表按 **分区（Partition）** + **分桶（Bucket）** 分布。
- **分区**：一般按时间（天/月），方便冷热数据管理。
- **分桶**：按分桶列哈希，将数据均匀散列到不同 BE。
- 多副本存在不同 BE，保证高可用。

# 三、存储引擎 & 数据模型
## 6. Doris 支持哪些数据模型？
**答**（必背）：
1. **Duplicate Key（重复键）**
   - 默认，原始数据存储，不聚合。
   - 适合日志、明细数据。
2. **Aggregate Key（聚合键）**
   - 导入时按 Key 预聚合（sum、replace、max、min）。
   - 适合报表、统计。
3. **Unique Key（唯一键）**
   - 保证 Key 唯一，自动去重/更新。
   - 适合需要**实时更新**的宽表。

## 7. Unique Key 实现原理？
**答**：
- 同一 Key 新数据覆盖旧数据。
- 底层通过 **合并（Merge）** 实现：
  - 读时合并（Read Merge）
  - 后台段合并（Segment Compaction）
- 保证**最终一致性**。

## 8. Doris 存储结构？
**答**：
- 列式存储。
- 数据按 **RowBatch → Segment → Tablet** 组织。
- 支持 **稀疏索引、min/max 索引、位图索引**。
- 后台自动 Compaction 合并小文件，提升查询性能。

# 四、索引 & 查询
## 9. Doris 有哪些索引？
**答**：
1. **前缀索引（Prefix Shortkey Index）**
   - 利用**排序列前缀**加速过滤，默认自带。
2. **ZoneMap 索引**
   - 每个数据块记录 min/max，自动跳过无关数据。
3. **Bitmap 索引**
   - 适合**低基数列**（性别、状态、地区、类型）。
4. **BloomFilter 索引**
   - 提高 IN/= 过滤效率。

## 10. 为什么 Doris 查询快？
**答**：
1. MPP 并行执行。
2. 列式存储 + 高压缩。
3. 向量化执行引擎。
4. 前缀索引、ZoneMap 大量剪枝。
5. 预聚合模型减少扫描数据量。
6. 算子下沉，尽可能在 BE 本地计算。

# 五、性能优化（面试必问）
## 11. Doris 查询慢如何优化？
**答**（标准八股）：
1. 设计合理 **分区 + 分桶**，按时间分区，高基数列分桶。
2. 设计合适 **ShortKey**，把高频过滤列放前面。
3. 大查询尽量**只查需要的列**。
4. 低基数列建 **Bitmap 索引**。
5. 使用 **合适模型**：明细用 Duplicate、聚合用 Aggregate、更新用 Unique。
6. 避免小文件过多，保证 **Compaction** 正常。
7. 合理设置并发，避免 FE/BE 资源打爆。
8. 大表 Join 尽量用 **Bucket Join / Colocate Join**。

## 12. 什么是 Colocate Join？
**答**：
- 将**相同分桶键**的表，数据**哈希在同一 BE 节点**。
- Join 时不需要数据网络传输，本地直接 Join，**极快**。
- 要求：分桶数相同、分桶列相同、Colocate Group 相同。

## 13. Doris 数据倾斜怎么解决？
**答**：
1. 选择**分布均匀的列**做分桶键。
2. 增加分桶数。
3. 对热点 key 加盐处理（加随机后缀）。
4. 使用 SQL 倾斜优化 hint。

# 六、数据写入 & 实时
## 14. Doris 写入方式有哪些？
**答**：
1. **Stream Load**：高吞吐本地文件导入。
2. **Routine Load**：订阅 Kafka 实时导入。
3. **Insert Into**：像 MySQL 一样插入。
4. **Broker Load**：读取外部存储（HDFS/S3）。
5. **MySQL Dump**：同步 MySQL 数据。

## 15. Doris 支持实时更新吗？
**答**：
支持，用 **Unique Key 模型**，可**实时更新、删除**，延迟秒级。
适合做**实时宽表、实时维度退化**。

# 七、和 ClickHouse 对比（高频）
## 16. Doris vs ClickHouse 对比？
**答**（面试最常用）：
- **Doris**
  - 架构简单，无外部依赖。
  - 支持 MySQL 协议，学习成本低。
  - **Join 性能更好**，支持复杂 SQL。
  - 更适合**实时数仓、高并发报表、多表关联**。

- **ClickHouse**
  - 单表查询、大宽表聚合**极快**。
  - 存储更极致，压缩比更高。
  - 不擅长多表复杂 Join。
  - 适合**日志分析、时序大宽表**。

一句话总结：
**复杂查询、多表关联、实时数仓选 Doris；单表超大表、简单聚合选 ClickHouse。**

# 八、运维 & 生产问题
## 17. BE 宕机影响？
**答**：
- 多副本架构，单个 BE 宕机不影响服务。
- FE 自动调度，副本提供服务。
- 修复后自动同步数据。

## 18. 数据导入成功但查不到？
**答**：
- 数据还在内存，未完成 **发布（Publish）**。
- 或 Compaction 未完成。
- 一般等待秒～分钟级可见。

## 19. 小文件太多怎么办？
**答**：
- 触发 Compaction 合并。
- 提高导入批次大小，减少小批次导入。
- 手动触发：`ALTER TABLE ... COMPACT`。

# 九、极简背诵版（直接背）
1. Doris 是 **MPP 架构的实时 OLAP 数据库**。
2. 架构：**FE（元数据+调度） + BE（存储+计算）**。
3. 数据分布：**分区+分桶**，多副本高可用。
4. 三大模型：**Duplicate、Aggregate、Unique**。
5. 索引：**前缀索引、ZoneMap、Bitmap**。
6. 优化核心：**分区、分桶、前缀索引、Bitmap、Colocate Join**。
7. 兼容 MySQL，**Join 强**，适合**实时数仓**。
8. 对比 ClickHouse：Doris 更擅长**复杂 SQL、多表关联、实时更新**。
