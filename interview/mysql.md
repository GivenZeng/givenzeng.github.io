
# 一、基础与架构（必问）
## 1. MyISAM 和 InnoDB 区别？（高频）
**答案：**
1. **事务**：MyISAM 不支持，InnoDB 支持
2. **锁**：MyISAM 表锁；InnoDB 行锁 + 表锁
3. **外键**：MyISAM 不支持，InnoDB 支持
4. **崩溃恢复**：MyISAM 易损坏，InnoDB 支持
5. **索引**：MyISAM 非聚簇；InnoDB **聚簇索引**
6. **并发**：InnoDB 高并发更好
**结论**：现在默认都用 InnoDB。

---

## 2. InnoDB 为什么用聚簇索引？
**答案：**
- 主键索引的叶子节点直接存**整行数据**
- 按主键查询**一次定位**，不用回表
- 范围查询、排序效率高

# 二、索引（必考核心）
## 3. 什么是 B+ 树？为什么不用二叉树/红黑树？
**答案：**
- B+ 树：**多路平衡查找树**，叶子节点形成有序链表
- 优势：
  - 树高度低（2~4 层），磁盘 IO 少
  - 范围查询、排序极快
  - 适合磁盘存储

## 4. 最左前缀原则
**答案：**
联合索引 `(a,b,c)`
能命中：
- a
- a,b
- a,b,c
不能命中：
- b
- b,c
- c
**必须从左开始连续匹配**。

## 5. 什么是回表？怎么避免？
**答案：**
- 通过普通索引查到主键，再去主键索引查数据 = 回表
- 避免：**覆盖索引**（查询字段都在索引里）

## 6. 索引失效场景（必考）
**答案：**
- 对索引列运算、函数、类型转换
- 模糊查询 `%前缀`
- not in / != / is not null
- 联合索引不满足最左前缀
- 优化器觉得全表更快（数据量小）

# 三、事务与锁（字节最爱问）
## 7. 事务四大特性 ACID
**答案：**
- A 原子性：要么全成功，要么全失败
- C 一致性：执行前后数据合法
- I 隔离性：事务间互不干扰
- D 持久性：提交后永久生效

## 8. 四大隔离级别 + 问题
**答案：**
1. 读未提交：脏读
2. 读已提交：不可重复读
3. 可重复读：**MySQL 默认**，解决脏读、不可重复读，可能幻读
4. 串行化：无问题，性能差

## 9. MVCC 是什么？原理？
**答案：**
多版本并发控制，用来**实现读写不冲突**。
- 每行隐藏 2 个字段：创建版本号、删除版本号
- 查询只看：
  - 创建版本 ≤ 当前版本
  - 删除版本未定义或 > 当前版本
**作用**：实现不加锁的非阻塞读。

## 10. 脏读、不可重复读、幻读区别
**答案：**
- 脏读：读到**未提交**的数据
- 不可重复读：同一事务两次查询结果**不一样**（update）
- 幻读：同一事务多次查询**条数不一样**（insert/delete）

## 11. InnoDB 锁类型
**答案：**
- 共享锁（S）：读锁，共享加锁
- 排他锁（X）：写锁，互斥
- 意向锁：表锁 + 行锁兼容
- 行锁：锁索引记录
- 间隙锁（GAP）：锁范围，防止幻读
- 临键锁（Next-Key Lock）：行锁 + 间隙锁，默认算法

## 12. 什么是死锁？怎么解决？
**答案：**
- 死锁：两个事务互相等待对方释放锁
- 解决：
  - 按**固定顺序**访问表
  - 小事务快提交
  - 设置锁超时
  - 开启死锁检测

# 四、SQL 优化（字节高频）
## 13. SQL 优化步骤
**答案：**
1. 看执行计划 `explain`
2. 看是否走索引
3. 优化 where、order by、join
4. 避免索引失效
5. 大SQL拆分，小事务

## 14. explain 关键字段
**答案：**
- type：最好到 ref、range，最差 ALL
- key：实际使用索引
- rows：扫描行数
- extra：Using filesort / Using temporary 必须优化

## 15. 为什么不建议用 select *
**答案：**
- 不需要的字段占用IO、内存
- 无法使用**覆盖索引**
- 回表概率变高
- 扩展表结构影响大

# 五、高可用、架构、主从（必问）
## 16. 主从复制原理
**答案：**
1. 主库写操作记录 binlog
2. 从库 IO 线程拉取 binlog 到 relay log
3. 从库 SQL 线程重放，同步数据

## 17. 主从延迟原因 & 解决
**答案：**
原因：
- 大事务、批量写入
- 从库配置低
- 从库单线程重放（低版本）
解决：
- 分库分表
- 开启并行复制
- 业务读主库（关键链路）

## 18. 分库分表场景与方案
**答案：**
场景：单库 TPS 过高、数据量超千万/亿
方案：
- 垂直分库：按业务拆分
- 水平分表：按用户ID/时间范围分片
分片键：选择高区分度、查询频繁字段
问题：跨分片查询、事务、分页

## 19. 分布式事务方案
**答案：**
- 2PC
- TCC
- 本地消息表
- 可靠消息最终一致性
- Seata

## 20. 海量数据分页优化
**答案：**
- `limit 1000000,10` 极慢
- 优化：
  - 延迟关联
  - 用主键范围 `where id > 1000000 limit 10`

# 六、高频压轴题（字节最爱）
## 21. MySQL 为什么有时候明明有索引却不走？
**答案：**
- 区分度低（如性别）
- 数据量小，优化器选择全表
- 隐式类型转换
- 使用函数、运算
- 联合索引不符合最左前缀

## 22. char 与 varchar 区别
**答案：**
- char：固定长度，速度快，浪费空间
- varchar：可变长度，省空间，需要计算长度

## 23. drop、truncate、delete 区别
**答案：**
- delete：DML，可回滚，删数据
- truncate：DDL，不可回滚，清空表，快
- drop：删表，不可恢复

## 24. 大表怎么加字段不锁表？
**答案：**
- 5.6+ 支持 Online DDL
- 用 gh-ost / pt-online-schema-change
- 低峰期执行

# 七、你必须背会的 10 句金句
1. InnoDB 是聚簇索引，MyISAM 非聚簇
2. 索引用 B+ 树，因为磁盘IO少
3. 联合索引遵循最左前缀
4. 覆盖索引避免回表
5. MVCC 实现无锁读
6. 可重复读是 MySQL 默认隔离级别
7. Next-Key Lock 解决幻读
8. 主从通过 binlog 同步
9. 大表优化优先分库分表
10. explain 看 type/key/rows/extra
