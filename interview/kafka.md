
# 一、基础原理（必问）
## 1. Kafka 是什么？核心组件？
**答案：**
分布式、高吞吐、低延迟的**消息队列/流处理平台**。
核心组件：
- Producer：生产者
- Consumer：消费者
- Broker：服务器节点
- Topic：消息主题
- Partition：分区（并发核心）
- Replica：副本（高可用）
- Zookeeper / KRaft：元数据管理

## 2. Kafka 为什么吞吐极高？
**答案：**
1. **Partition 分区**：并发读写
2. **顺序写磁盘**：避免随机 IO
3. **PageCache**：操作系统缓存，直接读写内存
4. **ZeroCopy 零拷贝**：减少内核/用户态拷贝
5. 批量发送、压缩消息

## 3. Partition 作用？为什么要多分区？
**答案：**
- 提高**并行度**：生产者、消费者、存储都能并发
- 一个 Topic 吞吐量由分区数决定
- 分区是 Kafka 并行和扩容的最小单位

## 4. Replica 副本作用？AR / ISR / OSR？
**答案：**
- Replica：保证高可用，一主多从
- AR：所有副本
- ISR：和 Leader 保持同步的副本集合
- OSR：不同步的副本
**只有 ISR 里的副本才能被选举为 Leader**

## 5. ISR 伸缩条件？
**答案：**
- 副本追上 Leader 日志偏移量
- 在 `replica.lag.time.max.ms` 内有请求
进入/退出 ISR 都由 Broker 自动维护。

# 二、生产者（高频）
## 6. Producer 发送消息流程？
**答案：**
1. 拦截器 → 序列化 → 分区器
2. 放入 **Batch 批次**
3. 由 Sender 线程发送到 Broker
4. 等待 ACK 确认

## 7. acks 参数含义？（字节必问）
**答案：**
- `acks=0`：发完就不管，吞吐量最高，丢数据
- `acks=1`：Leader 写入即成功，可能丢数据（Leader 宕机）
- `acks=-1/all`：Leader + ISR 全部同步完才成功，**最安全**

## 8. 如何保证消息不丢失？
**答案：**
- `acks=-1`
- `retries` 重试
- `unclean.leader.election.enable=false`
- 副本数 ≥ 2，ISR ≥ 2
- 生产者开启幂等

## 9. 什么是生产者幂等？原理？
**答案：**
开启 `enable.idempotence=true`，保证**消息不重复**。
原理：
- 每个 Producer 有 PID
- 每个消息有 Sequence Number
- Broker 去重

## 10. 分区策略有哪些？
**答案：**
- 指定 Partition
- Key 哈希取模
- 轮询 RoundRobin
- 粘性分区 Sticky Partition（默认，批量优化）

# 三、消费者 & 消费者组（必考）
## 11. 什么是消费者组？
**答案：**
- 一个 Group 内：**一个分区只被一个消费者消费**
- 多个 Group：互不影响，各自独立消费
作用：实现**广播**和**集群消费**。

## 12. 为什么消费者数不能大于分区数？
**答案：**
一个分区只能被同一个组内**一个消费者**消费。
消费者 > 分区数 → 多余消费者空闲，浪费资源。

## 13. 什么是重平衡（Rebalance）？触发条件？
**答案：**
Group 内协调消费分区分配的过程。
触发：
- 消费者加入/退出/超时
- 分区数变化
- Topic 被删除
**重平衡期间消费者不可用，要尽量避免**。

## 14. 消费位移（Offset）存哪里？
**答案：**
- 新版 Kafka：存内部主题 `__consumer_offsets`
- 不再依赖 ZK

## 15. 自动提交 vs 手动提交？
**答案：**
- 自动：简单，可能重复/丢失
- 手动：`commitSync` / `commitAsync`，业务可控，生产推荐

# 四、高可用 & 一致性（字节最爱）
## 16. Kafka 如何保证高可用？
**答案：**
- 多副本 Replica
- ISR 机制
- Leader 选举
- 故障自动转移

## 17. 什么是 HW、LEO？
**答案：**
- LEO：Log End Offset，副本最后一条消息偏移量
- HW：High Watermark，所有 ISR 都同步到的位置
**消费者只能读到 HW 之前的消息**，保证一致性。

## 18. 什么是数据一致性？怎么保证？
**答案：**
所有消费者看到的数据顺序一致。
保证：
- ISR 同步机制
- HW 水位线
- 只有 ISR 才能选举 Leader

## 19. 为什么不允许非 ISR 选举？
**答案：**
`unclean.leader.election.enable=false`
防止不同步副本被选为主，导致**数据丢失/不一致**。

# 五、存储机制 & 高吞吐（必问）
## 20. Kafka 文件存储结构？
**答案：**
Topic → Partition → Segment
每个 Segment 包含：
- `.log`：数据文件
- `.index`：偏移量索引
- `.timeindex`：时间索引

## 21. 零拷贝（ZeroCopy）原理？
**答案：**
使用 `sendfile`，数据直接从**PageCache → 网卡**，
不经过用户态，减少 2 次拷贝 + 2 次上下文切换。

## 22. Kafka 为什么顺序写？
**答案：**
磁盘随机 IO 很慢，顺序写接近内存速度，
Kafka 追加写，不修改历史数据。

# 六、生产问题（实战必问）
## 23. 消息丢失场景 & 解决？
**答案：**
1. 生产者：网络抖动 → 重试、幂等、acks=-1
2. Broker：Leader 宕机 + 非 ISR 选举 → 关闭 unclean
3. 消费者：自动提交 → 手动提交

## 24. 消息重复消费 & 解决？
**答案：**
原因：网络重试、提交失败、Rebalance。
解决：
- 业务**幂等**（唯一键、状态机）
- 生产者幂等 + 事务

## 25. 消息积压怎么解决？
**答案：**
1. 提高消费者线程数（≤ 分区数）
2. 增加分区数
3. 优化消费逻辑（减少 IO、批量、异步）
4. 优先消费关键数据
5. 临时扩容消费者

## 26. Rebalance 优化？
**答案：**
- 合理设置 `session.timeout.ms`、`max.poll.interval.ms`
- 消费要快，不要长事务
- 避免频繁启停消费者
- 使用**静态组员** Static Group Membership

# 七、高级特性（字节常考）
## 27. Kafka 事务原理？
**答案：**
保证**生产者多分区原子写入**，
或**消费-生产**原子性（Exactly-Once）。
依赖：
- TransactionID
- PID  epoch
- 事务协调者
- 事务日志主题

## 28. Exactly-Once 语义如何实现？
**答案：**
- 生产者幂等
- 事务
- 消费位移与生产消息原子提交

## 29. KRaft 对比 Zookeeper？
**答案：**
- KRaft：Kafka 内置一致性协议，替代 ZK
- 更少组件、更低延迟、更易运维
- 元数据直接存在 Broker，性能更高

# 八、字节最爱问的 10 句金句
1. 分区是 Kafka 并发与扩容的核心
2. ISR 是高可用与一致性的关键
3. acks=-1 才是生产安全配置
4. 消费者组实现广播/集群消费
5. 消费者数不建议超过分区数
6. Rebalance 会导致消费短暂不可用
7. HW 保证消费者看到一致数据
8. 顺序写 + PageCache + 零拷贝 = 高吞吐
9. 幂等 + 事务 = 精准一次语义
10. 消息积压优先加分区、提消费速度
